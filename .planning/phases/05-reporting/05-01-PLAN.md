<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Phase 4 execution artifacts -> reporting engine | The reporting engine may read execution outputs only after they exist and must reject incomplete or malformed artifacts. |
| reporting engine -> report artifacts | Report writes must be deterministic and run-scoped so replay cannot change business identity or backfill execution data. |
| report-node -> run status | The report node may advance the run to terminal completion, but it must not mutate manifest or execution-side state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | S/T | `src/guigui_strategy/reporting/node.py` input validation | mitigate | Validate that the report node only receives a run that already has the Phase 4 execution artifacts required by the report contract before synthesis begins. |
| T-05-02 | T/R | `src/guigui_strategy/reporting/persistence.py` artifact writes | mitigate | Use atomic writes into `{artifact_root}` and keep the report index deterministic so the same run inputs always yield the same report outputs. |
| T-05-03 | I | `report.md` and `reports/report-index.json` | accept | The report artifacts are intended to be readable evidence inside the run container; exposing them is required for audit and does not create an external sharing path. |
| T-05-04 | D | `src/guigui_strategy/reporting/engine.py` metric computation | mitigate | Bind all metrics to Phase 4 artifacts only, reject missing inputs, and avoid re-reading or inferring data from earlier phases during report synthesis. |
</threat_model>

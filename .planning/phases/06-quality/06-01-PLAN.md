<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Existing run container -> validation engine | The validation engine may read the run container only after it exists and must reject incomplete or mismatched artifacts. |
| validation engine -> meta/validation.json | Validation writes must be deterministic, run-scoped, and limited to the validation record; they must not backfill execution data. |
| validate entry -> manifest/status/execution artifacts | The validate entry may inspect and report on these artifacts, but it must not mutate manifest or execution-side state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | S/T | `src/guigui_strategy/validation/engine.py` input validation | mitigate | Validate that the run container has a consistent `run_id`, required top-level files, and the exact artifact layout expected by the contract before any deeper checks run. |
| T-06-02 | T/R | `src/guigui_strategy/validation/persistence.py` validation writes | mitigate | Use atomic writes into `{artifact_root}/meta/validation.json` and keep the record deterministic so the same run inputs always yield the same validation output. |
| T-06-03 | I | `meta/validation.json` and any derived validation summary | accept | Validation artifacts are intended to be readable evidence inside the run container; exposing them is required for audit and does not create an external sharing path. |
| T-06-04 | D | `src/guigui_strategy/validation/node.py` boundary behavior | mitigate | Bind validation to read-only inspection of the run container and reject attempts to repair, regenerate, or overwrite execution-side artifacts during validate. |
</threat_model>
